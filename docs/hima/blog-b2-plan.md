## 記事構成

### 1. 導入 — 「コピペ地獄」の描写（400 字）

AI チャットの繰り返し作業を具体的に描写する。

- EC 担当者が商品説明 50 件を 1 つずつ AI に書かせる風景
- 毎回: プロンプトをコピー → 商品名を差し替え → 結果をスプレッドシートに貼る
- 所要時間 30 分。AI のおかげで速くなったはずなのに、手作業は減っていない
- 「これ、人間の仕事じゃない。」

### 2. 既存ソリューションの限界（500 字）

なぜ既存ツールではこの問題が解けないのか。

- **ChatGPT / Claude**: 1 対 1 対話に最適化。バッチ処理の概念がない。50 件やるなら 50 回対話する
- **Zapier / Make**: AI ネイティブではない。セットアップが重い。プロンプトの試行錯誤に向かない
- **PromptChainer 系ツール**: サーバーサイド依存。データを外部に預ける必要がある。バッチ特化ではない
- 結論: 「バッチ × AI × クライアントサイド」の組み合わせが空白だった

### 3. Hima の設計判断（800 字）

4 つの設計判断を、それぞれ「なぜそうしたか」で語る。

**3-1. レシピ（プロンプトテンプレート）という概念**
- プロンプトを「使い捨て」から「資産」に変える
- `{{変数}}` で差し替え可能にし、再利用性を持たせた
- レシピは保存・共有・改良できる

**3-2. CSV を入力形式に選んだ理由**
- エンジニアもビジネス職も扱える共通言語
- Excel / Google スプレッドシートからのエクスポートが容易
- 行 = 1 タスク、列 = 変数。構造がバッチ処理と自然に対応する

**3-3. クライアントサイド完結の設計哲学**
- データがブラウザの外に出ない。サーバーに預けない
- プライバシーの問題を技術で解決する
- トレードオフ: サーバーサイドならキューイングやリトライが容易だが、ユーザーの信頼を優先した

**3-4. BYOK（Bring Your Own Key）を選んだ理由**
- ベンダーロックインの排除。OpenAI でも Anthropic でも Google でも
- ユーザーが自分の API キーを使う = Hima は中間マージンを取らない
- 完全無料が実現できる構造

### 4. 技術的な誠実さ（500 字）

隠さない。トレードオフを正直に語る。

- **CORS プロキシ**: ブラウザから直接 AI API を叩けない制約。最小限のプロキシを用意した。リクエストをリレーするだけで、データは保存しない
- **「完全クライアントサイド」の正確な意味**: API コールのリレー部分だけがサーバーを経由する。CSV データ・レシピ・実行結果はブラウザ内で完結
- **何を犠牲にしたか**: サーバーサイドキューイング、バックグラウンド実行、大規模バッチのスケーラビリティ
- **何を得たか**: ユーザーの信頼、プライバシー、運用コストゼロ

### 5. これから（300 字）

- ロードマップの方向性（レシピ共有マーケットプレイス、チーム機能など。詳細は別記事）
- コミュニティからのフィードバックが開発の優先度を決める
- X アカウントのフォロー、Hima の利用、フィードバック送信への誘導
