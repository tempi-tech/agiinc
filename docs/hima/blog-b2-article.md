## プロンプトをコピーして、貼って、また次のプロンプトをコピーする

EC サイトの商品説明を 50 件、AI に書いてもらう。やることはシンプルだ。

ChatGPT を開く。プロンプトを入力する。「以下の商品の説明文を 150 字で書いてください」。商品名を入れて送信。出力をコピーして、スプレッドシートの B2 セルに貼り付ける。

次。同じプロンプトをコピーして、商品名だけ差し替えて、送信。結果をコピーして、B3 セルに貼り付ける。

次。また同じ。B4、B5、B6……。これを 50 回繰り返す。所要時間 30 分。

AI のおかげで「書く」作業は圧倒的に速くなった。1 件あたりの文章生成は数秒。人間が書くより何倍も速い。でも「コピペする」作業は 1 ミリも速くなっていない。プロンプトをコピーし、商品名を差し替え、結果をスプレッドシートに転記する——この一連の手作業は、AI がどれだけ賢くなっても変わらない。

これが「コピペ地獄」だ。AI が進化するほど、繰り返し作業の虚しさが際立つ。せっかくの知能が、人間のコピペ速度で律速されている。

Hima はこの地獄を終わらせるために作った。

## 既存ツールでは、なぜ解けないのか

問題は新しくない。多くの人が同じ苦痛を感じている。ではなぜ、既存のツールで解決できていないのか。

**ChatGPT / Claude** — 優れた対話ツールだが、1 対 1 の対話に最適化されている。50 件処理するなら、50 回対話する。バッチ処理の概念がそもそも設計にない。

**Zapier / Make** — 自動化には強いが AI ネイティブではない。プロンプトの試行錯誤がしにくい。「温度パラメータを下げて再実行してみたい」「プロンプトを少し変えて比較したい」——そんな反復に向いていない。セットアップも重く、「ちょっと 10 件試したい」のハードルが高い。

**プロンプトチェーン系ツール** — AI ワークフローを組めるものはある。だが多くはサーバーサイド依存で、データを外部に預ける必要がある。バッチ処理に特化しているわけでもない。

整理すると、**「バッチ × AI × クライアントサイド」** という組み合わせに、ぽっかり空白があった。

Hima はその空白を埋めるために作った。

## 4 つの設計判断

Hima の形を決めたのは 4 つの判断だ。機能の話ではなく、「なぜそうしたか」の話をする。

### 1. レシピ — プロンプトを「使い捨て」から「資産」にする

AI チャットのプロンプトは使い捨てだ。30 分かけて良いプロンプトを練り上げても、チャット履歴に埋もれて二度と使われない。翌月また同じ作業が来たら、同じ試行錯誤をゼロからやり直す。

Hima ではプロンプトテンプレートを「レシピ」と呼ぶ。`{{商品名}}` `{{カテゴリ}}` のように二重波括弧で変数を埋め込み、CSV のカラムと対応させる。一度作ったレシピは名前を付けて保存し、ライブラリに蓄積する。次に同じ作業が発生したら、レシピを呼び出して CSV を差し替えるだけでいい。

プロンプトを「毎回書くもの」から「育てるもの」に変える。使うたびに改良し、精度を高めていく。チームで共有すれば、ノウハウが属人化しない。それがレシピの設計意図だ。

### 2. CSV — 「すでに持っているもの」を使う

入力形式に CSV を選んだのは意図的な判断だ。

CSV はエンジニアもビジネス職も扱える共通言語。Excel や Google スプレッドシートからワンクリックでエクスポートできる。新しいフォーマットを覚える必要がない。

そして CSV の構造はバッチ処理と自然に対応する。行が 1 タスク、列が変数。この対応関係が直感的だから、マッピング UI もシンプルに保てた。

JSON でも YAML でもなく CSV を選んだ理由は、**ユーザーの既存のワークフローに最も摩擦なく溶け込む**からだ。ツールに合わせてデータを変換する、あの手間を消したかった。

### 3. クライアントサイド完結 — データをブラウザの外に出さない

これが Hima の設計で最も強い制約であり、最も重要な判断だ。

ユーザーの CSV データ、レシピ、実行結果——すべてがブラウザの中で完結する。Hima のサーバーにデータが保存されることはない。API キーすらブラウザの localStorage にしか存在しない。

トレードオフは大きい。サーバーサイドならキューイングやリトライが容易だし、タブを閉じてもバッチを継続できる。バックグラウンド実行も、大規模バッチのスケーラビリティも確保しやすい。

それを承知の上で、クライアントサイドを選んだ。「あなたのデータは安全です」と言葉で約束するのではなく、**そもそもデータがサーバーに届かないという構造で保証する**。プライバシーの問題を、ポリシーではなく技術で解決する。

### 4. BYOK — 中間マージンを取らない

BYOK（Bring Your Own Key）は、ユーザーが自分の AI API キーを持ち込んで使う方式だ。

多くの AI ツールは API コールを自社サーバー経由で行い、利用料に上乗せする。Hima はこれをやらない。ユーザーの API キーで、プロバイダに直接つなぐ。OpenAI でも Anthropic でも Google でも、好きなモデルを選べる。

結果として Hima の利用は完全無料になる。かかるのは AI プロバイダへの API 利用料だけだ。ベンダーロックインもない。明日 OpenAI が新しいモデルを出しても、Google が価格を下げても、API キーを登録するだけですぐに使える。ユーザーは常に最新・最適なモデルを自分で選べる。

BYOK はユーザーとの関係を「依存」ではなく「共存」にするための判断だ。Hima がサービスを止めても、ユーザーのデータと API キーは自分の手元にある。その安心感を設計に織り込んだ。

## 正直に話す — 技術的なトレードオフ

「完全クライアントサイド」と聞くと万能に響くかもしれない。そうではない。隠さずに話す。

**CORS プロキシの存在。** ブラウザから AI プロバイダの API を直接叩くことはできない。API 側がブラウザからの直接アクセス（CORS）を許可していないからだ。そこで Hima は Cloudflare Worker を経由してリクエストを中継している。

この Worker が何をするか。ブラウザから受け取ったリクエストをそのまま AI プロバイダに転送し、レスポンスに CORS ヘッダーを付与して返す。それだけだ。データの保存もログの記録も一切しない。Worker のコードは極めてシンプルで、本質的にはヘッダーを 1 行追加するだけの中継役だ。

「完全クライアントサイド」の正確な意味を定義しておく。**CSV データ・レシピ・実行結果はブラウザ内で完結する。API コールのリレー部分だけが Cloudflare Worker を経由する。** Worker を通過するのはリクエストとレスポンスのみで、どこにも保存されない。

何を犠牲にしたか。サーバーサイドのキューイング、バックグラウンド実行、大規模バッチへのスケーラビリティ。タブを閉じれば結果は消える。

何を得たか。ユーザーの信頼、プライバシー、運用コストゼロの構造。

技術的な誠実さは、プロダクトの信頼と切り離せない。このトレードオフを隠すつもりはない。理解した上で使ってほしいから、こうして書いている。

## コピペ地獄を終わらせたかった。それだけだ。

50 回のコピペを、1 クリックに。

AI がせっかく賢くなったのに、人間がコピペ要員をやっている。その現状がどうしても気に入らなかった。

Hima はその答えとして設計した。レシピでプロンプトを資産にし、CSV で既存のワークフローに溶け込み、クライアントサイドでプライバシーを構造的に守り、BYOK でベンダーロックインを排除する。判断のひとつひとつに理由がある。完璧ではないが、正直に作った。
